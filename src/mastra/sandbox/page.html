Agentic RAG for Legal Cases

Ingest case documents into Pinecone, then iteratively search them with precision through the OpenClaw agent.

Overview

This skill adds document ingestion, case management, and agentic RAG search to the OpenClaw agent. Each legal case is an isolated Pinecone namespace. Documents are parsed, chunked, embedded, and stored with rich metadata so the agent can iteratively search, evaluate results, refine queries, and read full documents to give lawyers precise, citation-backed answers.

Key insight: Unlike one-shot RAG, agentic search loops up to 3 times — planning new queries, evaluating gaps, and narrowing down to the exact passage before reading the full document for final context.

Agentic Search Flow

When legal-agentic-search is invoked:

1
Query Planning Agent Legal Query Planner agent generates 3–5 diverse search queries using legal terminology
2
Vector Search Each query is embedded and searched against Pinecone (namespace = caseId), with optional party/document type filters
3
Result Evaluation Agent Legal Result Evaluator agent assesses specificity, identifies gaps (e.g. “found references to the timeline but not specific dates”)
4
Iterate or Proceed If gaps remain and iterations < 3, loop back to step 1 with refined queries. Otherwise, proceed.
5
Full Document Read Reads the full extracted text of the top candidate document from workspace/legal/cases/{caseId}/full-text/
6
Answer Synthesis Produces a precise, citation-backed answer referencing specific documents, page numbers, and quoted passages

Workflow: legal-rag

The Workflow version adds a human-in-the-loop clarification step before searching. It suspends to ask the lawyer 3 clarifying questions, then runs the same iterative search loop.

Using with OpenClaw

All legal RAG tools are registered on the OpenClaw agent. Here’s a typical conversation flow:

1. Create a case

"Create a new legal case called smith-v-jones-2024 for the case Smith v. Jones"

OpenClaw will call legal-create-case with caseId: "smith-v-jones-2024".

2. Ingest documents

"Ingest the deposition transcript at /path/to/smith-deposition.pdf into the
smith-v-jones-2024 case. It's a deposition, and the relevant parties are
John Smith, Jane Jones, and Acme Corp."

OpenClaw will call legal-ingest-document with the file path, case ID, document type, and party names.

3. Batch ingest

"Ingest all of these into smith-v-jones-2024:
  - /docs/exhibit-a.pdf (exhibit, parties: Acme Corp)
  - /docs/contract.docx (contract, parties: John Smith, Acme Corp)
  - /docs/financials.xlsx (discovery, parties: Acme Corp)"

4. Search

"In the smith-v-jones-2024 case, what did John Smith say about
the timeline of the contract amendment?"

OpenClaw recognizes this as a complex query and uses legal-agentic-search. It will iterate through search queries, evaluate results, and read the full deposition transcript before giving a precise answer with page references.

5. Quick lookup

"Find the indemnification clause in the smith-v-jones-2024 case"

For simple lookups, OpenClaw will use legal-case-search for a fast one-shot search.

Scaling Beyond 100+ Documents

Current design: Optimized for 10–25 documents per case. For 100+ documents per case, consider these adjustments:
  • Chunking strategy: Switch from recursive (512 tokens) to semantic-markdown or sentence for better semantic boundaries
  • Embedding batching: The embedMany call handles batching automatically, but monitor rate limits with large corpora
  • Metadata filters: Lean on documentType and partyNames filters to narrow search scope before vector similarity
  • Re-ranking: Add Cohere or LLM-based re-ranking to the search pipeline via createVectorQueryTool’s reranker option
  • Parallel ingestion: Use legal-batch-ingest but consider queueing with concurrency limits for very large batches
  • Full-text storage: For very large documents (>50k chars), the agentic search truncates to 50k. Consider storing section-level extracts instead
